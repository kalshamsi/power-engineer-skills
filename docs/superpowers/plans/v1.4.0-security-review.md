# v1.4.0 Hook Scripts — Security Review

**Auditor:** Opus security-review subagent (separate dispatch from per-task Dual-Opus auditors)
**Date:** 2026-04-17
**Scope:** `power-engineer/scripts/hooks/session-end-handoff.sh` + `power-engineer/scripts/hooks/pre-compact-snapshot.sh`
**Threat model:** Shell injection · path traversal · unsafe destructive commands · env var assumptions · error handling · Power Engineer-specific (hostile project dir / git output / race conditions)
**Methodology:** Line-by-line static review + ShellCheck (incl. `--include=SC2016` to verify suppressions) + runtime exploitation tests with hostile inputs (malicious `CLAUDE_PROJECT_DIR`, malicious commit messages with `$(…)` and backticks, branch-name newline injection, `state_dir`-exists-as-regular-file fault injection, paths with spaces).

---

## Summary

- 🚨 Critical: 0
- ⚠️ Minor: 2
- ✅ Pass: 14

**Verdict:** RELEASE-SAFE

Both scripts are well-defended: `set -euo pipefail` everywhere, all variables consistently double-quoted, no `eval` / no untrusted command substitution, an `ERR` trap that logs and exits 0, and an `EXIT` trap that guarantees exit 0 on every termination path. Untrusted git output (commit messages, branch names) is **interpolated as printf data arguments**, never as command tokens. ShellCheck passes clean with the documented SC2016 suppressions, all 6 of which are confirmed false positives (literal markdown backticks in single-quoted format strings — verified by removing the disables and re-running shellcheck, which then reports exactly the 6 expected info-level warnings on the lines marked, none anywhere else).

Runtime exploitation tests confirm the hardening: feeding a commit message of literal `$(touch /tmp/PWNED1)` produces a markdown file containing the *literal text* `$(touch /tmp/PWNED1)` — `/tmp/PWNED1` is never created. Feeding `CLAUDE_PROJECT_DIR='$(touch /tmp/PWNED2)'` likewise produces no execution; it is treated as a (legitimate) directory name.

Two minor items are noted for v1.4.1 hardening but do not block release: a theoretical timestamp-collision file-overwrite if two hooks fire within the same UTC second, and absence of any pre-flight validation that `$state_dir` is actually a directory (the hook silently swallows the resulting `mkdir` and write errors and exits 0, which is the *correct* hook contract behavior, but means a misconfigured user environment could produce no handoff and only a single line in `memory-errors.log`).

### SC2016 suppression review

**SC2016** (info-level): "Expressions don't expand in single quotes, use double quotes for that."

The script disables this on lines 92 and 95. Both target identical printf calls:

```bash
printf '```\n%s\n```\n\n' "$recent_commits"
printf '```\n%s\n```\n\n' "$modified_files"
```

The triple-backtick literals (`` ``` ``) inside the single-quoted format string are markdown code-fence delimiters, not bash command substitution. ShellCheck has no way to know the script is generating markdown.

**Verification:** I removed both `# shellcheck disable=SC2016` lines and re-ran `shellcheck` on the modified copy. Output:

```
In /tmp/test-session-no-disable.sh line 92:
    printf '```\n%s\n```\n\n' "$recent_commits"
           ^----------------^ SC2016 (info): Expressions don't expand in single quotes, use double quotes for that.

In /tmp/test-session-no-disable.sh line 94:
    printf '```\n%s\n```\n\n' "$modified_files"
           ^----------------^ SC2016 (info): Expressions don't expand in single quotes, use double quotes for that.
```

The warnings target *only* the literal-backtick segments and *only* on the lines flagged. No other SC2016 warnings appear elsewhere in the script. The suppressions are factually correct and surgically scoped.

The variables `$recent_commits` and `$modified_files` are correctly *outside* the single-quoted format string — they are double-quoted positional arguments to printf, supplied as data for the `%s` placeholder.

### Findings

- ✅ `set -euo pipefail` plus ERR/EXIT traps guarantee the "always exit 0" hook contract is honored on every termination path, including unhandled errors.
- ✅ All untrusted inputs (commit messages, branch names, file paths from git) are routed through `printf '%s' "$var"` patterns where the variable is a *data argument*, never a format string. Confirmed safe against `$(…)`, backticks, `$VAR`, `\n`, and `%n`-style format-string attacks.
- ✅ `CLAUDE_PROJECT_DIR` is properly defaulted to `$PWD` and consistently double-quoted at every use site, preventing word-splitting under `set -u` even with hostile values.
- ✅ The `git_field` subshell pattern correctly isolates `cd` from the parent shell.
- ✅ Runtime test: hostile commit message `$(touch /tmp/PWNED1)` did NOT execute. The string was written as literal text into the markdown file.
- ✅ Runtime test: `CLAUDE_PROJECT_DIR='$(touch /tmp/PWNED2)'` did NOT execute. A directory with that literal name was created (correct behavior for a literal arg) and the markdown was written into it.
- ✅ Runtime test: paths containing spaces (`/tmp/hook test 2/with spaces`) work correctly.
- ✅ Runtime test: when `.power-engineer` exists as a *regular file*, the script logs an error to `memory-errors.log` and exits 0 — does not corrupt user state.
- ⚠️ MINOR: `handoff_file` uses second-resolution timestamps (`%Y%m%dT%H%M%SZ`). Two SessionEnd hook invocations within the same UTC second would write to the same filename and clobber. Practically impossible in normal Claude Code use (SessionEnd fires once per session terminator), but a malicious or buggy harness firing it in a tight loop would lose one handoff per collision. **Recommendation for v1.4.1:** append `$$` (PID) or `$RANDOM` to the timestamp.
- ⚠️ MINOR: When `mkdir -p "$state_dir"` fails (e.g., a regular file at that path, a permission denial, or a read-only filesystem), the user gets a single line in `memory-errors.log` and no handoff. There is no surfacing path back to the operator beyond manually reading that log. This is the **correct** behavior per the hook contract (must exit 0, must not block session termination), so it is a UX/observability gap, not a security flaw. **Recommendation for v1.4.1:** consider documenting in the user-facing CHANGELOG/MIGRATION that operators should periodically `cat .power-engineer/memory-errors.log` to surface silent hook failures.

## Recommendations

### Address before release (🚨 CRITICAL)

None. No release-blocking findings.

### Address in v1.4.1 (⚠️ MINOR)

1. **Timestamp-collision file overwrite.** Both scripts compute output filenames using only second-resolution UTC timestamps. Two hook invocations within the same UTC second would clobber. Append `$$` (PID) or `$RANDOM` for collision avoidance, e.g.:
   ```bash
   handoff_file="${state_dir}/session-handoff-${timestamp_filename}-$$.md"
   ```
   Practical risk: very low under normal Claude Code use. Worth fixing for defensive hygiene.

2. **Operator observability for silent hook failures.** When `mkdir -p` fails or the write block fails, the user sees nothing (exit 0 is the *correct* hook contract). The only failure trail is `.power-engineer/memory-errors.log`. Consider:
   - Adding a one-line note to MIGRATION.md or CHANGELOG.md telling operators to periodically check `memory-errors.log`.
   - Optionally: at the start of each hook invocation, count log-file lines and `echo` a warning to stderr (Claude Code may surface this in the transcript) if the count is non-zero. Trade-off: more noise vs. better observability. Defer to user preference.

### No action required (✅ PASS)

- Strict-mode shell (`set -euo pipefail`) plus ERR + EXIT traps correctly enforce the "always exit 0" hook contract.
- All untrusted inputs are interpolated via `printf '%s'` data arguments; no command substitution of attacker-controlled strings; no `eval`; no unquoted variables in command position.
- `CLAUDE_PROJECT_DIR` is correctly defaulted with `${VAR:-default}` syntax and double-quoted everywhere.
- The 6 `SC2016` disable directives are factually false positives, narrowly scoped to single-quoted printf format strings containing literal markdown backticks. Verified by removing each disable and re-running shellcheck — exactly the expected info-level warnings appear, none anywhere else.
- Runtime exploitation tests confirmed:
  - Malicious commit message `$(touch /tmp/PWNED1)` does NOT execute.
  - Malicious `CLAUDE_PROJECT_DIR='$(touch /tmp/PWNED2)'` does NOT execute.
  - Newline injection in branch names is rejected by git itself before reaching the hook.
  - When `.power-engineer` exists as a regular file, the script silently logs and exits 0 — does not crash, does not corrupt.
  - Paths containing spaces are handled correctly.
- Both scripts are byte-identical in their security-critical infrastructure (env resolution, error sink, traps, `git_field`, write block); reviewing one is effectively reviewing both for those subsystems.

**Final verdict:** RELEASE-SAFE
