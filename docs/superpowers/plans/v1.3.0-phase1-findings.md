# v1.3.0 Phase 1 — Findings & Informational Items

During Phase 1 (Lint Foundation) of the v1.3.0 upgrade (commits 00a5dea..2cf6a65 on branch `v1.3.0-upgrade`), four pre-existing repository states surfaced that the plan had not anticipated. All were addressed within Phase 1 scope or deferred with user acknowledgment. This document preserves the findings for downstream phases (especially Phase 2 catalog refactor and Phase 9 release notes).

## 1. Catalog has five schema shapes in active use

The Phase 1 plan assumed catalog tables come in two shapes: the canonical 5-column form and a 3-column legacy form. The actual catalog contains five distinct header patterns spread across `docs/superpowers/catalog/`. When `scripts/catalog-integrity.sh` was written, it was broadened to accept all five rather than fail on unexpected columns.

Accepted shapes (all currently valid):
- `| Skill | Install | Description | Trigger | When to use |` — canonical 5-col (`ai-coding.md`, `devops.md`, most files)
- `| Skill | Scope | Install | Description | Trigger | When to use |` — 6-col with Scope (`anthropic-official.md`)
- `| Skill | Source | Install | Description | Trigger | When to use |` — 6-col with Source (`engineering/data-ml.md`)
- `| Skill | Source | Scope | Install | Description | Trigger | When to use |` — 7-col with Source+Scope (`docs-research.md`)
- `| Suite | Trigger | When to use |` — legacy 3-col (`power-suites.md`; install commands live in fenced blocks below each table row, not inline)

**Phase 2 implication:** The `@latest` strip script must handle all five column layouts. A regex keyed on column position will silently miss the 6-col and 7-col Install cells if it assumes Install is always column 2.

## 2. power-suites.md schema is different from what the plan described

The plan described `power-suites.md` as a standard catalog table with an Install column. In reality the file uses `| Suite | Trigger | When to use |` (3 columns) and places install commands in fenced code blocks immediately below each table row. There is no inline Install cell to strip.

This changes two Phase 2 tasks. First, the `@latest` strip pass must also scan fenced blocks in `power-suites.md` — a pure table-column substitution will leave all `@latest` tags in that file untouched. Second, the count-verification step in Phase 2 Task 2.2 Step 3 needs to count both inline Install cells across the 4-col through 7-col files *and* `npx` lines inside fenced blocks in `power-suites.md` to arrive at a correct total.

## 3. `source` field dropped from installed_skills schema

Historical versions of `.power-engineer/state.json` stored a `source` field on each installed skill entry, with values `"power-engineer"` or `"manual"`. That field has been replaced by a single `installed_by` field. The drop is not recorded in git history — the change predates the current commit range and was discovered during Phase 1 schema validation work.

Phase 1 added a negative-guard check to `scripts/catalog-integrity.sh` that fails if a `source` key is found in `installed_skills`, preventing accidental re-addition. If Phase 9's `MIGRATION.md` documents schema changes between v1.2.x and v1.3.0, this field rename should be listed as a breaking change for any tooling that reads `state.json` directly.

## 4. README badge skill-count drift

The README badge declares **238 skills**. A row count of the catalog markdown files yields **224 data rows** — a gap of 14. The discrepancy is explained by deliberate cross-category duplicates: `firecrawl` appears in both `docs/superpowers/catalog/docs-research.md` and `engineering/data-ml.md`; `skill-creator` and `mcp-builder` each appear twice in `anthropic-official.md`. The badge was never updated to reflect removals or deduplication.

Phase 8's `scripts/update-skill-count.sh` is the designated fix for this. However, if Phase 2 normalizes catalog rows and removes or consolidates duplicates, the opportunity to reconcile the badge count earlier is worth taking — it avoids the badge being wrong through five more phases.
