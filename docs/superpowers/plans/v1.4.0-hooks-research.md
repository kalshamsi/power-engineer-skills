## Executive Summary

Research against the live official Claude Code documentation at `code.claude.com` (the current canonical domain — `docs.anthropic.com/en/docs/claude-code/hooks` now 301-redirects there) reveals a substantially richer hook system than common training-data assumptions reflect. The critical findings are:

1. **Schema shape is definitively nested** — `{type, command}` is correct and confirmed. The hook registration structure is two levels deep: `hooks.EventName` → array of matcher-groups → each matcher-group has a `matcher` string and a `hooks` array of handler objects. This matches the v1.3.0 fix pattern and is not flat.

2. **SessionEnd exists but is NOT fully reliable** — it was introduced in v1.0.85 but community-confirmed GitHub issues (#17885, #6428) show it does **not** fire on `/exit` slash command (closed as "not planned") and reportedly does not fire on `/clear` despite documentation listing "clear" as a valid reason value. Implementers must treat SessionEnd as best-effort, not guaranteed.

3. **PreCompact CAN block compaction** — confirmed in CHANGELOG v2.1.105: "hooks can now block compaction by exiting with code 2". This is a significant capability for the v1.4.0 implementation.

4. **SessionStart matcher `"compact"` is confirmed current** — the current settings.json already uses `"matcher": "compact"` under `SessionStart`, and the docs confirm accepted values are `"startup"`, `"resume"`, `"clear"`, and `"compact"`. The v1.3.0 bug (PostToolUse with `"compact"` matcher) arose from using the wrong event type, not a wrong matcher value.

5. **`settings.local.json` has HIGHER precedence than `settings.json`** for scalar settings (Local overrides Project overrides User). For hooks specifically, precedence does NOT apply — all matching hooks from all sources merge and run in parallel, not cascade.

## Schema Shape Verification (settings.json)

The **exact** current schema from `code.claude.com/docs/en/hooks` (fetched 2026-04-16):

```json
{
  "hooks": {
    "EventName": [
      {
        "matcher": "string|pattern|*",
        "hooks": [
          {
            "type": "command|http|prompt|agent",
            "command": "path/to/script.sh",
            "timeout": 600,
            "statusMessage": "custom spinner message",
            "once": false,
            "async": false,
            "asyncRewake": false,
            "shell": "bash"
          }
        ]
      }
    ]
  },
  "disableAllHooks": false
}
```

**Key structural facts (all sourced from `code.claude.com/docs/en/hooks`):**

- The outer `hooks` object maps **event name strings** to arrays.
- Each event's array contains **matcher-group objects** (not handler objects directly).
- Each matcher-group has:
  - `matcher` — string or regex pattern controlling when this group fires (see per-hook detail below).
  - `hooks` — inner array of **handler objects**, each with `type` + handler-specific fields.
- This two-level nesting (`hooks.Event[].hooks[]`) is the **current canonical shape**. A flat structure (placing `type`/`command` directly in the event array) is not documented and was the source of the v1.3.0 regression.

**Example command handler — full field reference:**

```json
{
  "type": "command",
  "command": "\"$CLAUDE_PROJECT_DIR\"/.claude/hooks/my-hook.sh",
  "timeout": 600,
  "statusMessage": "Running hook...",
  "once": false,
  "async": false,
  "asyncRewake": false,
  "shell": "bash"
}
```

Source: `https://code.claude.com/docs/en/hooks` — "Command Hook Fields Reference" section.

## Hook-by-Hook Reference

### SessionStart

- **Availability:** Yes — documented and confirmed working.
- **Introduced:** Prior to v1.3.0 (present in repo's own `.claude/settings.json` baseline).
- **Firing:** Fires when Claude Code starts a new session OR resumes an existing session. Fires once per session. Confirmed synchronous by default (blocks session start until hook completes, or until timeout).
- **Context/stdin JSON:**
  ```json
  {
    "session_id": "abc123",
    "transcript_path": "/Users/.../.claude/projects/.../transcript.jsonl",
    "cwd": "/path/to/project",
    "permission_mode": "default",
    "hook_event_name": "SessionStart",
    "source": "startup|resume|clear|compact"
  }
  ```
  The `source` field distinguishes the trigger type.
- **Matcher values (exact strings accepted):**
  | Value | When it fires |
  |-------|--------------|
  | `"startup"` | New session (claude invoked fresh) |
  | `"resume"` | Session resumed (`--resume` or `--continue`) |
  | `"clear"` | After `/clear` command clears the conversation |
  | `"compact"` | After `/compact` compaction event |
  | `""` or `"*"` or omitted | Always — all SessionStart events |
- **Special capability — `CLAUDE_ENV_FILE`:** SessionStart (and CwdChanged, FileChanged) hooks can persist environment variables. If `CLAUDE_ENV_FILE` is set, appending `export VAR=value` to that file makes the variable available in all subsequent Bash commands for the session.
- **Example registration block:**
  ```json
  {
    "hooks": {
      "SessionStart": [
        {
          "matcher": "compact",
          "hooks": [
            {
              "type": "command",
              "command": "\"$CLAUDE_PROJECT_DIR\"/.claude/hooks/post-compact-restore.sh"
            }
          ]
        },
        {
          "matcher": "startup",
          "hooks": [
            {
              "type": "command",
              "command": "\"$CLAUDE_PROJECT_DIR\"/.claude/hooks/load-context.sh"
            }
          ]
        }
      ]
    }
  }
  ```
  Source: `https://code.claude.com/docs/en/hooks` — Example 3 + `https://code.claude.com/docs/en/settings`.
- **Gotchas:**
  - CHANGELOG v2.1.73: "Fixed SessionStart hooks firing twice when resuming a session via `--resume` or `--continue`". This was a bug — ensure you are on v2.1.73+ if using `resume` matcher.
  - The current repo `.claude/settings.json` correctly uses `SessionStart` with `"matcher": "compact"`. This is valid and documented. The v1.3.0 bug was using `PostToolUse` with `"compact"` (wrong event entirely).
- **Source URLs:**
  - `https://code.claude.com/docs/en/hooks`
  - `https://code.claude.com/docs/en/settings`

### SessionEnd

- **Availability:** Yes — introduced in v1.0.85, but **partially undocumented and incompletely implemented**. The GitHub issue #6306 (filed 2025, closed without official response) noted it was missing from docs; the current `code.claude.com/docs/en/hooks` page now lists it. However, known bugs persist.
- **Introduced:** v1.0.85 per GitHub issue #6306 reference.
- **Firing:** Fires when a session terminates. **DOES NOT always fire** — confirmed community bugs:
  - **Does NOT fire on `/exit` command** — GitHub issue #17885 (filed 2026-01-13, closed as "not planned" with "stale" label). ctrl-D exit fires it; `/exit` does not.
  - **Does NOT fire on `/clear` command** per GitHub issue #6428 (open bug, "has repro" label) despite docs listing `"clear"` as a valid reason value.
  - CHANGELOG v2.1.75: "Fixed SessionEnd hooks being killed after 1.5s on exit regardless of `hook.timeout`" — timeout bug was fixed.
  - CHANGELOG v2.1.80: "Sessions now execute end-of-conversation hooks when users exit or switch conversations" — additional fix pass.
- **Context/stdin JSON:**
  ```json
  {
    "session_id": "abc123",
    "transcript_path": "/path/to/transcript.jsonl",
    "cwd": "/path/to/project",
    "permission_mode": "default",
    "hook_event_name": "SessionEnd",
    "reason": "clear|resume|logout|prompt_input_exit|bypass_permissions_disabled|other"
  }
  ```
  The `reason` field is the event-specific addition.
- **Matcher values (exact strings):**
  | Value | When it fires |
  |-------|--------------|
  | `"clear"` | `/clear` command (BUGGED — may not fire, see above) |
  | `"resume"` | Session paused for resume |
  | `"logout"` | User logged out |
  | `"prompt_input_exit"` | Exit via prompt input |
  | `"bypass_permissions_disabled"` | Permissions bypass disabled |
  | `"other"` | Catch-all for other termination reasons |
  | `""` / `"*"` / omitted | All SessionEnd events |
- **Can hook block termination?** No. SessionEnd is a fire-and-observe event; it cannot prevent session termination. Exit code 2 does not block exit. Source: `https://code.claude.com/docs/en/hooks` — "SessionEnd | No | Shows stderr to user only".
- **Synchronous vs async:** Synchronous by default (blocks for `timeout` duration). Post-v2.1.75 fix ensures timeout is respected.
- **Example registration block:**
  ```json
  {
    "hooks": {
      "SessionEnd": [
        {
          "hooks": [
            {
              "type": "command",
              "command": "power-engineer/scripts/hooks/session-end-handoff.sh"
            }
          ]
        }
      ]
    }
  }
  ```
  Note: matcher omitted = fires on all SessionEnd reasons (recommended for session-handoff use case).
- **Gotchas:**
  - **Best-effort only** — `/exit` command does not trigger it. Users who exit via `/exit` (the most common keyboard shortcut) will not get the hook. Treat SessionEnd as a supplement to, not a replacement for, explicit memory saves.
  - The `reason` field value `"exit"` appears in some third-party examples but is NOT in the official documented list. Use `"other"` to catch unclassified exits.
  - CLAUDE.md's proactive memory rules remain the primary reliability guarantee; SessionEnd is a bonus automation layer.
- **Source URLs:**
  - `https://code.claude.com/docs/en/hooks`
  - `https://github.com/anthropics/claude-code/issues/17885`
  - `https://github.com/anthropics/claude-code/issues/6306`
  - `https://github.com/anthropics/claude-code/issues/6428`

### PreCompact

- **Availability:** Yes — documented and confirmed. Blocking capability added in v2.1.105.
- **Introduced:** Prior to v2.1.105 (blocking capability added in that version).
- **Firing:** Fires before Claude Code runs a compaction operation, either triggered automatically (context fills up) or manually by the user via `/compact`. This is a **pre-flight** hook — it fires before the compaction summary is generated, giving you a chance to preserve information.
- **Context/stdin JSON:**
  ```json
  {
    "session_id": "abc123",
    "transcript_path": "/path/to/transcript.jsonl",
    "cwd": "/path/to/project",
    "permission_mode": "default",
    "hook_event_name": "PreCompact",
    "trigger": "manual|auto",
    "custom_instructions": "any /compact instructions provided by user"
  }
  ```
  The `trigger` and `custom_instructions` fields are event-specific. No compaction statistics or target-size information is provided in the input — it is a simple pre-flight notification.
- **Matcher values:**
  | Value | When it fires |
  |-------|--------------|
  | `"manual"` | User explicitly ran `/compact` |
  | `"auto"` | Automatic compaction triggered by context limit |
  | `""` / `"*"` / omitted | Both manual and auto compaction |
- **Can hook BLOCK compaction?** Yes — confirmed in CHANGELOG v2.1.105: "Added PreCompact hook support: hooks can now block compaction by exiting with code 2." Exit code 2 is the documented blocking exit code across all blockable hooks.
- **Synchronous:** Yes — fires synchronously. The compaction waits for the hook to complete (or timeout) before proceeding.
- **Example registration block:**
  ```json
  {
    "hooks": {
      "PreCompact": [
        {
          "matcher": "auto",
          "hooks": [
            {
              "type": "command",
              "command": "power-engineer/scripts/hooks/pre-compact-snapshot.sh"
            }
          ]
        }
      ]
    }
  }
  ```
  Source: `https://claude.com/blog/how-to-configure-hooks` — PreCompact example section. Confirmed against `code.claude.com/docs/en/hooks`.
- **Gotchas:**
  - No compaction statistics are available in the hook input — you cannot inspect how much will be summarized.
  - If you want the hook to run for both manual and auto compaction (recommended for snapshot use case), omit the matcher or use `"*"`.
  - Exit code 2 blocks compaction — use with care; blocking auto-compaction in a long session could cause context overflow errors.
- **Source URLs:**
  - `https://code.claude.com/docs/en/hooks`
  - `https://claude.com/blog/how-to-configure-hooks`
  - `https://raw.githubusercontent.com/anthropics/claude-code/refs/heads/main/CHANGELOG.md` (v2.1.105 entry)

### SubagentStop

- **Availability:** Yes — documented as part of the async events group. Fires independently of the main session loop.
- **Introduced:** Referenced in CHANGELOG v2.1.92 in context of Stop/SubagentStop hook behavior fixes.
- **Firing:** Fires when a subagent completes its task. This is an async event — it fires independently without blocking the main session.
- **Context/stdin JSON:**
  ```json
  {
    "session_id": "abc123",
    "transcript_path": "/path/to/transcript.jsonl",
    "cwd": "/path/to/project",
    "permission_mode": "default",
    "hook_event_name": "SubagentStop",
    "agent_id": "agent-abc123",
    "agent_type": "Explore"
  }
  ```
  The `agent_id` and `agent_type` fields are subagent-specific.
- **Matcher values:**
  The matcher filters by agent type. Accepted values:
  | Value | Description |
  |-------|-------------|
  | `"Bash"` | Built-in Bash subagent |
  | `"Explore"` | Built-in Explore subagent |
  | `"Plan"` | Built-in Plan subagent |
  | Custom agent names | Names from `.claude/agents/` |
  | Regex patterns | e.g., `"^Explore"` or `".*-agent"` |
  | `"*"` / omitted | All subagent types |
  Source: `https://code.claude.com/docs/en/hooks` — "SubagentStart, SubagentStop | agent type | same values as SubagentStart".
- **Can hook block?** Not explicitly stated for SubagentStop. Since it's listed in the async events group, it fires fire-and-forget.
- **Note on skill/agent frontmatter:** The docs state "In subagent frontmatter, `Stop` hooks are automatically converted to `SubagentStop` since that event fires when the subagent completes." This is relevant for hooks embedded in skill frontmatter.
- **Example registration block:**
  ```json
  {
    "hooks": {
      "SubagentStop": [
        {
          "matcher": "Explore",
          "hooks": [
            {
              "type": "command",
              "command": ".claude/hooks/log-subagent-done.sh"
            }
          ]
        }
      ]
    }
  }
  ```
- **v1.4.0 relevance:** NOT being used in v1.4.0 per plan spec. Documented here for completeness.
- **Source URLs:**
  - `https://code.claude.com/docs/en/hooks`
  - `https://raw.githubusercontent.com/anthropics/claude-code/refs/heads/main/CHANGELOG.md` (v2.1.92 entry)

### Stop

- **Availability:** Yes — documented and confirmed.
- **Firing:** Fires when Claude finishes responding (end of a turn). This is a per-turn event — it fires once per user prompt cycle, after Claude has completed its response.
- **Context/stdin JSON:**
  ```json
  {
    "session_id": "abc123",
    "transcript_path": "/path/to/transcript.jsonl",
    "cwd": "/path/to/project",
    "permission_mode": "default",
    "hook_event_name": "Stop"
  }
  ```
  No event-specific fields beyond the common input fields. Source: `https://code.claude.com/docs/en/hooks`.
- **Matcher values:** Stop does NOT support matchers. The docs explicitly list Stop in the "no matcher support" group: "UserPromptSubmit, Stop, TeammateIdle, TaskCreated, TaskCompleted, WorktreeCreate, WorktreeRemove, CwdChanged — don't support matchers. Added matchers silently ignored."
- **Can hook block?** Yes. Exit code 2 prevents Claude from stopping and continues the conversation. Decision pattern uses top-level `decision` field:
  ```json
  {
    "decision": "block",
    "reason": "explanation"
  }
  ```
- **Synchronous:** Yes by default. Can be made async with `"async": true`.
- **Example registration block:**
  ```json
  {
    "hooks": {
      "Stop": [
        {
          "hooks": [
            {
              "type": "command",
              "command": ".claude/hooks/on-turn-complete.sh"
            }
          ]
        }
      ]
    }
  }
  ```
  Note: matcher field in the outer group object should be omitted (it is silently ignored).
- **Gotchas:**
  - Matcher field is silently ignored — do not add a `"matcher"` to Stop hook groups. This is a common mistake that produces no error but confuses readers of the settings.
  - CHANGELOG v2.1.92: "Fixed prompt-type Stop hooks incorrectly failing when the small fast model returns `ok:false`" — ensure you are on v2.1.92+.
  - `StopFailure` is a separate event that fires when a turn ends due to an API error. If you need to handle errors specifically, use StopFailure.
- **v1.4.0 relevance:** NOT being used in v1.4.0. Documented here for completeness.
- **Source URLs:**
  - `https://code.claude.com/docs/en/hooks`

## settings.json vs settings.local.json Precedence

From `code.claude.com/docs/en/settings` — "How scopes interact" section (quoted verbatim):

> When the same setting is configured in multiple scopes, more specific scopes take precedence:
>
> 1. **Managed** (highest) — can't be overridden by anything
> 2. **Command line arguments** — temporary session overrides
> 3. **Local** — overrides project and user settings
> 4. **Project** — overrides user settings
> 5. **User** (lowest) — applies when nothing else specifies the setting

**File mapping:**

| Scope | File | Shared? |
|-------|------|---------|
| User | `~/.claude/settings.json` | No |
| Project | `.claude/settings.json` | Yes (committed to git) |
| Local | `.claude/settings.local.json` | No (gitignored) |

**Critical nuance for hooks:** Unlike scalar settings where higher precedence overrides lower ones, **hooks merge across all sources and run in parallel**. The docs state: "All hooks from all sources merge and run in parallel." There is no override/replace mechanism for hooks between files — they accumulate.

**Decision conflict resolution:** When multiple hooks return conflicting decisions on the same event, the precedence is: `deny` > `defer` > `ask` > `allow`.

**Implication for Task 2.7:** Modifying the project-level `.claude/settings.json` (which this repo uses as its tracked config) adds hooks that run **in addition to** any user-level (`~/.claude/settings.json`) or local (`settings.local.json`) hooks. They do not replace each other. If a developer has a conflicting hook in their user settings, both will run.

Source: `https://code.claude.com/docs/en/settings` — "Configuration scopes" and "How scopes interact" sections.

## Hook-Script File Path Conventions

From `code.claude.com/docs/en/hooks` — "Environment Variables for Hook Scripts" section:

| Variable | Resolves to | Example |
|----------|-------------|---------|
| `$CLAUDE_PROJECT_DIR` | Project root | `/home/user/my-project` |
| `${CLAUDE_PLUGIN_ROOT}` | Plugin installation directory | Updated on plugin refresh |
| `${CLAUDE_PLUGIN_DATA}` | Plugin persistent data directory | Survives plugin updates |

**Path conventions:**
- The docs use `"$CLAUDE_PROJECT_DIR"/.claude/hooks/script.sh` as the standard example path — referencing scripts **within** the `.claude/hooks/` directory of the project.
- There is **no hard requirement** for scripts to live in `.claude/hooks/`. Absolute paths and paths relative to `$CLAUDE_PROJECT_DIR` both work.
- The v1.4.0 plan specifies `power-engineer/scripts/hooks/session-end-handoff.sh` and `power-engineer/scripts/hooks/pre-compact-snapshot.sh` — paths under the skill tree so `npx skills add` ships them to end users (per Phase 2 relocation commit `d747e07`). This is a valid architectural choice (enables ShellCheck + unit-testability). The command in settings.json should reference these as either absolute paths or via `$CLAUDE_PROJECT_DIR`:
  ```json
  "command": "\"$CLAUDE_PROJECT_DIR\"/power-engineer/scripts/hooks/session-end-handoff.sh"
  ```
- Scripts must be **executable** (`chmod +x`). Non-executable scripts will fail silently (or with a permission error, depending on OS).
- **No `.claude/hooks/` discovery convention** is documented — Claude Code does not auto-discover scripts in that directory. Every hook script must be explicitly registered in settings.json.

Source: `https://code.claude.com/docs/en/hooks`.

## Implementation Guidance for Phase 2

### For Task 2.2 — SessionEnd hook registration in configurator.md

- **Use the nested schema** confirmed above. The registration block is:
  ```json
  "SessionEnd": [
    {
      "hooks": [
        {
          "type": "command",
          "command": ".claude/hooks/session-end-handoff.sh"
        }
      ]
    }
  ]
  ```
  Omit the `matcher` field to catch all SessionEnd reasons (since the hook's job is session handoff, you want it to run on any exit path).
- **Do NOT treat SessionEnd as guaranteed.** Due to the `/exit` command bug (issue #17885, closed as not-planned) and `/clear` bug (issue #6428, open), the hook will silently not fire in common exit scenarios. The fallback contract (CLAUDE.md auto-memory rules + explicit `save-phase`) is not optional — it is the primary reliability layer. SessionEnd is a bonus when it fires.
- **Set a reasonable timeout.** Default is 600s; for a session-handoff script that writes a file, 30–60s is sufficient. Add `"timeout": 60` to the handler.
- **Script must be executable** at the path referenced.

### For Task 2.3 — PreCompact hook registration in configurator.md

- **PreCompact CAN block compaction** (exit code 2). The snapshot script should NOT exit 2 unless there is a genuine reason to block — unexpected blocking will cause context overflow in long sessions.
- **Omit the matcher** to capture both `"manual"` and `"auto"` compaction events. If you want to differentiate behavior, use two separate matcher-group objects.
- **No compaction statistics** are available in the hook input. The script has access to `transcript_path` (the full transcript JSONL) and can read it for snapshot purposes.
- **`custom_instructions`** is available in the input JSON if the user passed instructions to `/compact`.
- Registration block:
  ```json
  "PreCompact": [
    {
      "hooks": [
        {
          "type": "command",
          "command": "power-engineer/scripts/hooks/pre-compact-snapshot.sh"
        }
      ]
    }
  ]
  ```

### For Task 2.7 — Installing hooks on repo's own `.claude/settings.json`

- **Read the existing settings.json first** (already done as part of this research). Current state has `SessionStart` with `"compact"` matcher — this is correct and should be preserved.
- **Add SessionEnd and PreCompact entries** under the existing `"hooks"` key. Do NOT replace the `SessionStart` entry.
- **Hooks merge across files**, so user-level settings won't conflict — both sets will run in parallel. This is safe.
- **Use `$CLAUDE_PROJECT_DIR`-relative paths** or absolute paths for the command field. Since the scripts ship at `power-engineer/scripts/hooks/session-end-handoff.sh` inside the skill tree (relocated in Phase 2 commit `d747e07` so `npx skills add` copies them to end-user machines):
  ```json
  "command": "power-engineer/scripts/hooks/session-end-handoff.sh"
  ```
  (Relative paths work if Claude Code is invoked from the project root, which is the standard case. For safety, prefer `"$CLAUDE_PROJECT_DIR"/power-engineer/scripts/hooks/...`.)
- **Verify scripts are executable** before committing: `test -x power-engineer/scripts/hooks/session-end-handoff.sh`.
- **settings.local.json precedence note:** If any developer has local overrides in `.claude/settings.local.json`, those override scalar settings in `.claude/settings.json`, but hooks from both files run in parallel. No action needed.

### General do's and don'ts

| Do | Don't |
|----|-------|
| Use nested `{matcher, hooks: [{type, command}]}` shape | Use flat `{type, command}` directly in the event array |
| Exit with code 0 (success) or 2 (blocking error) | Use exit code 1 — it is NOT blocking (non-blocking error only) |
| Quote paths with `"$CLAUDE_PROJECT_DIR"` in JSON | Use unquoted shell paths that expand incorrectly in JSON |
| Set `"timeout"` on scripts that should complete quickly | Rely on the 600s default for short-running scripts |
| Keep shell profiles quiet (no stdout on init) | Let shell startup scripts print to stdout (corrupts JSON parsing) |
| Treat SessionEnd as best-effort | Treat SessionEnd as a guaranteed save mechanism |
| Test hook scripts produce valid JSON on stdout | Output debug text without suppressing it (corrupts hook JSON) |
