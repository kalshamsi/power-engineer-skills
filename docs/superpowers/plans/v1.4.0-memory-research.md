## Executive Summary

**Recommendation: Build** — no candidate from the Claude Code skills marketplace meets the 90% alignment threshold for wholesale adoption, and MCP memory servers address a fundamentally different problem (external persistence layers) than the 3-tier in-process hook system power-engineer requires. The strongest Section 1 candidate (`grandamenium/dream-skill`) achieves ~55% alignment: it covers the PreCompact-equivalent tier well but uses a Stop-hook-with-time-check auto-trigger (not SessionEnd), has no explicit 4-type memory semantics (user / feedback / project / reference), and its consolidation model (sleep-dream-prune) does not match power-engineer's phase-level ceremony design. The strongest Section 2 pattern — mem0's external-storage approach to compaction-immunity — is transferable as an architectural principle but requires no third-party service. The Build path stands: implement all 3 tiers fresh, borrowing the "index cap enforcement" pruning pattern from dream-skill and the "namespace scoping" pattern from mem0/supermemory.

### Per-Candidate Detail

#### Investigation 1: mem0 (mem0ai/mem0)

**GitHub Stars:** ~37,000–50,000 (widely cited at 48k+ as of 2025–2026; most widely adopted agent memory framework)
**Last Active:** Active (frequent releases; OpenMemory initiative launched recently)

**Storage model:**
- Hybrid: vector database (default: Qdrant, Chroma, or Pinecone for semantic storage) + optional graph database (Neo4j for mem0g variant) + key-value for structured data
- OpenMemory variant: fully local, on-machine storage — no cloud required

**Retrieval strategy:**
- Primary: semantic search via embedding model (default: OpenAI `text-embedding-3-small`; compatible with Qwen 600M+ for local)
- Hybrid: semantic + keyword + entity boosting for best results
- Auto-Recall: on each turn, relevant memories are automatically injected into working context

**Compaction approach:**
- Architectural insight: external storage = **compaction-immune**. Memories live outside the LLM context window; compaction/truncation cannot affect them.
- On context growth, Auto-Recall re-injects only relevant memories on each new turn — memory is not replayed verbatim but retrieved on demand
- This completely sidesteps the context-growth problem rather than managing it

**Namespace / tenant model:**
- First-class scoping by `user_id`, `agent_id`, and `run_id` (session ID)
- Multi-tenant by design; each scope is isolated
- Supports filtering: retrieve only memories for a specific user + agent + session combination

**MCP compatibility:**
- Official MCP server: `mem0ai/mem0-mcp` (Python, pip-installable)
- Community implementations: `coleam00/mcp-mem0`, `pinkpixel-dev/mem0-mcp`
- OpenMemory MCP Server: compatible with any MCP client; fully local storage

**Transferable patterns for power-engineer's Build:**
1. **Compaction-immunity via external storage**: power-engineer's PreCompact hook (Tier 3) should snapshot working state to flat-file before compaction; the hook fires explicitly and the files persist outside the context window — same compaction-immune property without requiring mem0
2. **Scoped retrieval on demand**: power-engineer's SessionStart / restore behavior should load only the relevant project memory (scoped by project dir), not all memories globally — matches mem0's `user_id + agent_id` scoping pattern
3. **MEMORY.md ≤200 line index cap**: mem0 injects only *relevant* memories, not all memories. Power-engineer should follow the same discipline: the index is a lean pointer list; verbosity lives in topic files

#### Investigation 2: Zep / Graphiti (getzep/graphiti)

**GitHub Stars:** Graphiti: ~14,000 (8 months ago); crossed 20,000 stars by April 2026 (35+ contributors, 25,000 weekly PyPI downloads)
**Last Active:** Active (production deployment; arxiv paper published Jan 2025)

**Storage model:**
- Temporal knowledge graph: nodes = entities (people, orgs, concepts), edges = relationships with `valid_at` + `invalid_at` timestamps
- Backed by Neo4j (graph DB); also compatible with local graph stores
- Facts are versioned: "I live in NYC" → "I moved to SF" correctly supersedes the prior fact via edge invalidation

**Retrieval strategy:**
- Hybrid: semantic similarity (embedding) + keyword BM25 + graph traversal (relationship paths)
- Temporal queries: retrieve facts valid at a specific point in time, or how a relationship evolved over time
- Significantly outperforms MemGPT on Deep Memory Retrieval (DMR) benchmark per arxiv paper

**Compaction approach:**
- Graph persistence is entirely external to the LLM context window — immune to compaction
- Retrieval extracts a subgraph relevant to the current query and injects it into context on demand
- Knowledge remains stable across compaction events

**Namespace / tenant model:**
- Graph namespacing by agent_id, user_id, session entities
- Temporal scoping: queries can be bounded to a time window

**MCP compatibility:**
- Official MCP server: `getzep/graphiti/mcp_server/` — Graphiti Knowledge Graph MCP Server
- Available for Claude Desktop, Raycast, note-taking apps, and any MCP-compatible client
- Described as "persistent, local graph memory for any MCP client"

**Transferable patterns for power-engineer's Build:**
1. **Temporal fact versioning**: power-engineer's 4-type memory files should include a `lastUpdated` frontmatter field; when an existing memory is updated, the previous entry is preserved with a date range rather than overwritten silently. This is weaker than a full knowledge graph but captures the spirit.
2. **Hybrid retrieval concept**: the CLAUDE.md auto-memory rules that load memory files at SessionStart should prioritize by recency (temporal) + project relevance (structural), not just alphabetical order
3. **Relationship-awareness**: when saving project memories, linking related memories by reference (e.g., a `session_handoff` referencing the `project_v140_brainstorm` entry) improves recall quality — matches Graphiti's edge-based linking

#### Investigation 3: Supermemory (supermemoryai/supermemory)

**GitHub Stars:** ~13,500 (main repo); MCP variant separately maintained
**Last Active:** Active (March 2026 updates; Claude Code plugin: `supermemoryai/claude-supermemory`)

**Storage model:**
- Hybrid: RAG document store + stateful facts extraction layer
- Built on Cloudflare Workers + Durable Objects (scalable, persistent connections)
- Extracts facts from conversations, tracks updates, resolves contradictions, auto-forgets expired information
- Ranks #1 on LongMemEval, LoCoMo, and ConvoMem benchmarks

**Retrieval strategy:**
- Hybrid: RAG (document retrieval) + memory facts (structured facts about user/project)
- Automatically runs both: RAG for general context, facts layer for personalization
- Contradictions resolved: newer facts supersede older ones (e.g., "I just moved to SF" overrides "I live in NYC")

**Compaction approach:**
- Container-tag namespace isolation: memories are scoped to a project tag via `x-sm-project` header
- When context is compacted, the next turn re-injects relevant memories via Auto-Recall — compaction-immune architecture (same principle as mem0)
- "The Supermemory plugin preserves conversation context across compaction events"

**Namespace / tenant model:**
- Container tags: each project gets a unique container tag (auto-generated via containerTagPrefix + hash, or explicit)
- Separates work/personal context, client repos, arbitrary groupings
- Cross-context memories possible but scoped by default

**MCP compatibility:**
- Official MCP server: `supermemoryai/supermemory-mcp` ("Universal Memory MCP")
- Claude Code plugin: `supermemoryai/claude-supermemory` — explicitly targets Claude Code (real-time learning, updating, growing)
- No-login, no-paywall claim for open-source MCP tier

**Transferable patterns for power-engineer's Build:**
1. **Container-tag project isolation**: power-engineer's memory files are already scoped by `~/.claude/projects/<proj>/memory/`; this is architecturally equivalent to Supermemory's container-tag model. The pattern validates power-engineer's current filesystem-namespace approach.
2. **Contradiction resolution**: when save-phase writes a new `project` memory that updates a prior entry, the memory writing logic should check the MEMORY.md index for existing entries on the same topic and UPDATE rather than APPEND — already called out in CLAUDE.md's proactive memory rules, but Supermemory's explicit "resolve contradictions" step reinforces formalizing this as a check in the Tier 1 save-phase flow.
3. **Fact extraction as a separate pass**: Supermemory extracts "facts" from raw conversation, separate from storing full documents. This is the equivalent of power-engineer's memory type taxonomy — structured extraction over raw verbatim saving.

#### Investigation 4: Letta / MemGPT (letta-ai/letta)

**GitHub Stars:** ~20,000–30,000 (Apache 2.0; built by 100+ contributors; exact count not surfaced but project is prominent in the memory agent space)
**Last Active:** Active (v1.x releases in 2025–2026; Letta Code product launched for coding agents specifically)

**Storage model:**
- OS-inspired tiered memory:
  - Main context / working memory (in-window): what the agent currently "knows"
  - Archival storage (external, long-term): searchable, persisted outside context window
  - Recall storage (episodic): conversation history
- Agent self-edits its own memory using tool calls (`memory_insert`, `memory_replace`, `archival_memory_insert`, `archival_memory_search`)

**Retrieval strategy:**
- Agent-initiated: the agent decides when to search archival storage using tool calls
- Semantic search over archival storage
- Main context is always in-window; archival requires explicit retrieval

**Compaction approach:**
- Explicit compaction with overflow protection (fixes for double-compaction, runaway loops in recent releases)
- Default context window raised from 32k to 128k
- Agent autonomously manages what stays in main context vs. gets promoted/demoted to archival
- Context compaction = sliding window truncation with agent-managed memory preservation

**Namespace / tenant model:**
- Namespacing by `agent_id` + `conversation_id`; multiple agents can coexist
- `conversation_id="default"` supported for simpler use cases

**MCP compatibility:**
- Native MCP support via `mcp_config.json` integration
- Letta Code product specifically targets coding agents (memory-first coding agent)

**Transferable patterns for power-engineer's Build:**
1. **Explicit tier boundaries**: Letta's main-context / archival model maps cleanly to power-engineer's 3 tiers. The principle: Tier 1 (save-phase) = explicit promotion to archival. Tier 2 (SessionEnd) = sweep of remaining in-window working state to archival. Tier 3 (PreCompact) = emergency snapshot of main context before truncation.
2. **Agent-self-editing memory**: the save-phase flow should give Claude the explicit instruction to UPDATE existing memory entries rather than always append-only — matching Letta's `memory_replace` tool call semantics. This prevents MEMORY.md bloat.
3. **Graceful compaction overflow**: the PreCompact hook script should include a size check on the snapshot file and truncate to a safe maximum if the snapshot itself would exceed a limit — preventing the hook output from being larger than the compacted context.

## Final Recommendation

**Recommendation: Build**

**Rationale:**
No Section 1 candidate meets the 90% adoption threshold. The highest-scoring candidate (`grandamenium/dream-skill`) scores 55% — well below threshold. Its alignment gaps are structural: it uses a Stop hook (not SessionEnd), lacks a PreCompact tier, and has no 4-type memory taxonomy. Adopting it wholesale would require gutting its hook architecture, rewriting its consolidation model to match power-engineer's phase ceremony, and adding the 4-type frontmatter system — at which point it is no longer an adoption, it is a Build with dream-skill as a reference.

**Why not Adapt (borrow a skill wholesale with modifications)?**
Adapt is the middle path when a candidate covers 60–89% of the requirement and only needs targeted extensions. Dream-skill is architecturally close but the two missing tiers (SessionEnd, PreCompact) are the core of Phase 2's value proposition. Adapting dream-skill's shell script would still require writing all 3 tier scripts from scratch and would introduce a dependency on an unmaintained (small community) repo's design decisions. The risk/benefit does not favor Adapt over Build.

**What alternatives were rejected and why:**
- `hanfang/claude-memory-skill` (40%): no hook automation at all; manual-only invocation disqualifies it for Tier 2/3 responsibilities
- `cwinvestments/memstack` (25%): commercial Pro tier, 112-skill overhead, no 4-type taxonomy
- `anthropics/skills`: no memory skill exists; Anthropic's in-progress auto-dream feature is unreleased and not in this repo
- `obra/superpowers`: no memory skill; community skills repo archived Oct 2025
- `mattpocock/skills`: no memory skill

**MCP memory servers (Section 2) conclusion:** All 4 investigated systems (mem0, Zep/Graphiti, Supermemory, Letta) serve a different architectural purpose — they are external persistence layers accessed via API or MCP protocol, requiring infrastructure (vector DB, graph DB, cloud workers, or a running server process) that is incompatible with power-engineer's design constraint of zero external dependencies and graceful degradation to flat-file Markdown. None should be Adopted.

**Consequences for Phase 2 scope:** Build path confirmed; Phase 2's 3-tier implementation plan stands unchanged. The following patterns from Section 2 investigations are incorporated into the Build as design inputs (not dependencies):

1. **From mem0**: Compaction-immunity via external storage → PreCompact hook (Tier 3) writes snapshot to flat-file before compaction fires; this file survives the compaction event
2. **From Zep/Graphiti**: Temporal fact versioning → Tier 1 save-phase should include `lastUpdated` in frontmatter and check for existing entries on the same topic before writing new ones
3. **From Supermemory**: Container-tag project isolation → validates power-engineer's `~/.claude/projects/<proj>/memory/` namespace model; contradiction resolution formalizes CLAUDE.md's existing "update rather than duplicate" rule
4. **From Letta**: Explicit tier boundaries + agent-self-editing memory → Tier 1 save-phase flow instructs UPDATE-or-APPEND decision; PreCompact hook includes snapshot size guard
