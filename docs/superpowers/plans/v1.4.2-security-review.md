## Summary

- 🚨 Critical: **0**
- ⚠️ Minor: **0**
- ✅ Pass: **15** verified threat axes (8 inherited + 7 net-new)

**Overall: RELEASE-SAFE**

`scripts/catalog-diff.sh` is well-defended at every shell-injection boundary. All git refs supplied via CLI args are passed through `git rev-parse --verify --quiet --end-of-options "${ref}^{commit}"` BEFORE any `git show`/`git ls-tree` invocation. Refs that look like flags (`--upload-pack=…`, `-h`, `-rf`), contain command substitution (`$(…)`, backticks), contain newlines, semicolons, or pipes are all uniformly rejected with exit 2 and the error message "Error: ref '<...>' not found" — no shell evaluation of the hostile string occurs. The script contains zero `eval`, zero `jq` invocations (JSON is hand-emitted via printf with single-quoted format strings + `%s` data placeholders), and all command substitutions use literal-only commands (`mktemp`, `wc -l`, `tr`, `git diff` with quoted refs).

The `^[a-z0-9][a-z0-9-]*$` regex used by `uninstall.md` Step 2 and `info.md` Step 2 correctly rejects every tested attack pattern (path traversal, slashes, command substitution, backticks, spaces, uppercase, leading hyphen, real and literal newlines, semicolons, pipes, dot-segments, empty string). The `uninstall.md` execution block adds a second defense-in-depth layer — `case "$TARGET" in "$SAFE_ROOT"/*) rm -rf -- "$TARGET" ;; *) refuse ;; esac` — which would block path escape even if a hypothetical regex bypass existed. The `--` argument in `rm -rf --` further hardens against any future ref containing a leading hyphen reaching `rm` as a flag.

The `.github/workflows/ci.yml` `catalog-version-sync` and `catalog-diff-self-test` jobs do NOT interpolate any `github.event.*` data into shell command position. The only `${{ … }}` substitution in step bodies is `${{ secrets.GITHUB_TOKEN }}` passed via `env:` (the GitHub-recommended safe pattern). No PR-controlled data (branch name, commit message, PR title, author name) reaches `bash -c` arg position. ShellCheck 0.11.0 reports zero warnings on both `scripts/catalog-diff.sh` and `tests/lint/catalog-diff.sh` at default severity AND with the strict-mode disable list (`-e SC1090,SC2086,SC2046`) requested in the dispatch.

12 runtime exploitation tests against `scripts/catalog-diff.sh` (5 documented in the dispatch + 7 additional belt-and-suspenders cases) all rejected hostile input with exit 2 and the controlled "Error: ref '...' not found" message. Zero PWNED marker files were created on the filesystem. The script's existing self-test (`tests/lint/catalog-diff.sh` Test 5) covers the canonical `$(rm -rf /)` rejection path and is wired into the new `catalog-diff-self-test` CI job that runs on every PR.

## Net-new attack surface in v1.4.2

The genuinely new surface introduced this release is:

1. **`scripts/catalog-diff.sh`** — accepts user-supplied git refs from CLI args and feeds them to `git show`/`git ls-tree`/`git diff` after validation. The `validate_ref` function (line 116-123) is the load-bearing safety boundary; it uses `git rev-parse --verify --quiet --end-of-options "${ref}^{commit}"` which is the canonical defense pattern (recommended by the git project itself for pre-validating refs from untrusted sources). 12/12 exploit attempts rejected.

2. **`.github/workflows/ci.yml` `catalog-diff-self-test` job** — runs `tests/lint/catalog-diff.sh` on every PR. The self-test consumes only hard-coded args + the checkout. PR-controlled data does NOT reach script invocation. The job is bounded by GitHub's per-PR ephemeral runner — even if the script were compromised, the blast radius is one ephemeral runner with no `secrets.*` access (no `env:` block on the job other than the implicit `GITHUB_TOKEN` which is not exported in this job).

3. **`uninstall.md` Step 4 destructive operation** — the only `rm -rf` in user-facing flows. Defense layers: (a) Step 2 regex validation rejects every metachar/path-traversal attempt; (b) Step 4 SAFE_ROOT constructor builds the path from a hard-coded prefix + already-validated name; (c) Step 4 case-statement guard re-checks the constructed path begins with SAFE_ROOT before invoking `rm -rf --`; (d) the `--` argument prevents flag interpretation. Triple-redundant — even a hypothetical regex bypass would be caught by the prefix check.

The remaining v1.4.2 changes are doc-only (`uninstall.md`, `info.md` are flow specs read by Claude — not executable shell), test-runner additions (skip behavioral fixtures), CI workflow rewires (script body change, no new secret/data flow), and release artifacts (CHANGELOG, MIGRATION, README). None introduce executable attack surface.

## CI workflow safety analysis

`.github/workflows/ci.yml` was modified to:

1. Replace the `catalog-version-sync` step body with `./scripts/catalog-diff.sh --ci-check` (lines 79-89). Job name preserved for CI history continuity.

**Audit:**

- Both jobs use `actions/checkout@v4` with `fetch-depth: 0` to enable `origin/main...HEAD` ref comparisons.

- Neither job exports `GITHUB_TOKEN` to a step body (the only `secrets.*` use in the workflow is in `url-check-targeted` and `url-check-full`, unchanged from prior releases).

The only `${{ … }}` substitutions across the entire workflow are the two `secrets.GITHUB_TOKEN` env assignments (lines 46, 65) — the GitHub-recommended safe pattern (env vars, not script interpolation).
- A hostile PR cannot influence the script's behavior beyond submitting catalog-file modifications, which is exactly the surface the `--ci-check` job is designed to police. The script reads files from the checkout and emits an exit code; no mutation of the runner environment.

**Conclusion:** No CI-level escalation surface. The new jobs have the same security posture as the existing `lint` and `badge-sync` jobs.

**RELEASE-SAFE.**

The single cross-cutting concern worth noting (NOT a finding, just a forward-looking observation): the `validate_ref` defense relies on `git rev-parse --verify --quiet --end-of-options` continuing to be the canonical safe-validation pattern. If git itself ships a CVE in `rev-parse --verify --quiet --end-of-options` ref handling, every script that depends on this pattern (including catalog-diff.sh) would inherit the exposure. This is true of every shell script that pre-validates refs and is not specific to v1.4.2; ongoing git security advisory monitoring is the appropriate response, not script-level changes.
