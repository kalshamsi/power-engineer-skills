## Per-question answers

### 1. `git show <ref>:path` semantics

**Claim verified:** `git show "v1.4.0:power-engineer/.catalog-version"` prints the file content as it existed at that tag. Failure modes return exit 128 with `fatal:` on stderr.

**Evidence (live commands):**

| Test | Command | Exit | Output |
|------|---------|------|--------|
| Valid tag, valid path | `git show "v1.4.0:power-engineer/.catalog-version"` | 0 | `1.4.0` (file content) |
| Nonexistent ref | `git show "nonexistent-tag-xyz:power-engineer/.catalog-version"` | 128 | `fatal: invalid object name 'nonexistent-tag-xyz'.` |
| Valid ref, nonexistent path | `git show "v1.4.0:nonexistent/path.md"` | 128 | `fatal: path 'nonexistent/path.md' does not exist in 'v1.4.0'` |
| Path exists locally but not at ref | `git show "v1.2.0:power-engineer/.catalog-version"` | 128 | `fatal: path 'power-engineer/.catalog-version' exists on disk, but not in 'v1.2.0'` |

**Citations:**
- `git-show(1)` synopsis: "Shows one or more objects (blobs, trees, tags and commits)... For plain blobs, it shows the plain contents."
- `gitrevisions(7)`: *"A suffix `:` followed by a path names the blob or tree at the given path in the tree-ish object named by the part before the colon."*
- `gitrevisions(7)` defines `<rev>:<path>` syntax explicitly; this is THE canonical spelling for "read file at ref" without checking out.

**Recommendation:** The script should always invoke `git show "${ref}:${path}"` with both arguments double-quoted as a single composite string. Suppress stderr (`2>/dev/null`) when probing for optional content — but **NOT** when validating user input — and check the exit code explicitly so that fatal errors don't propagate as empty output.

### 2. Ref validation idiom

**Claim verified:** Both `git rev-parse --verify --quiet "$ref^{commit}"` and `git cat-file -e "$ref^{commit}"` reject hostile input safely (because they're called with double-quoted args — no shell expansion happens at all). However, **`git rev-parse --verify --quiet --end-of-options "$ref^{commit}"` is the recommended idiom**, sourced directly from the `git-rev-parse(1)` examples.

**Evidence (live tests with hostile input):**

| Hostile input | `rev-parse --verify --quiet` | `cat-file -e` |
|----|----|----|
| `$(echo HACKED >&2)` | exit 1, silent | exit 128, `fatal: Not a valid object name $(echo HACKED >&2)^{commit}` |
| `` `echo PWNED` `` | exit 1, silent | exit 128, `fatal: Not a valid object name`...echoes the backticks |
| `v1.4.0; rm -rf /tmp/...` | exit 1, silent | exit 128, `fatal: ...` echoes the ref |
| `--something-malicious` (flag injection) | exit 1, silent (with `--end-of-options`) | exit 128, echoes ref |

**No shell expansion ever happens** because both idioms are called with the user input inside double quotes. The command-substitution-looking text passes through to git as a literal string and git rejects it.

**Comparison:**

| Property | `rev-parse --verify --quiet --end-of-options` | `cat-file -e` |
|----------|---------|---------|
| Hostile input safety | ✅ (with `--end-of-options`) | ✅ (no flag injection vector since no flags consumed) |
| Exit code on invalid ref | 1 | 128 |
| Stderr on invalid ref | silent (with `--quiet`) | verbose `fatal:` line that ECHOES the user input |
| Stderr verbosity preference | ✅ silent — script can emit a custom user-friendly error | ❌ echoes hostile input verbatim into stderr (minor info-leak in log files; aesthetic noise) |
| Performance (1000 calls) | 4.64s | 4.60s (tied — no meaningful difference) |
| Documented in git man page | ✅ exact idiom in `git-rev-parse(1)` examples | ❌ not documented for ref-validation use |

**Citations:**
- `git-rev-parse(1)` examples section: *"Print the commit object name from the revision in the $REV shell variable: `$ git rev-parse --verify --end-of-options $REV^{commit}` This will error out if $REV is empty or not a valid revision."*
- `git-rev-parse(1)` `--quiet` flag: *"Only meaningful in `--verify` mode. Do not output an error message if the first argument is not a valid object name; instead exit with non-zero status silently."*

**Recommended idiom (use this verbatim):**

```bash
validate_ref() {
    # Validate user-supplied git ref. Returns 0 if valid, 1 otherwise.
    # Uses --end-of-options to defend against flag injection (e.g. ref
    # starting with --), --quiet to suppress git's stderr so we can
    # emit a user-friendly error message instead.
    local ref="$1"
    if ! git rev-parse --verify --quiet --end-of-options "${ref}^{commit}" >/dev/null; then
        printf 'Error: %q is not a valid git ref reachable from this repo\n' "$ref" >&2
        return 1
    fi
}

# Usage:
validate_ref "$ref_a" || exit 1
validate_ref "$ref_b" || exit 1
```

**Note on `printf '%q'`:** `%q` shell-escapes the user input when echoing it back in the error message — defends against terminal escape sequence injection if hostile input contained ANSI codes.

### 3. `jq` parse + key-presence

**Claim verified:** `jq -e` exits 1 if and ONLY IF the LAST output value is null or false. Comma-listing keys (`.a, .b, .c, .d, .e`) is **UNSAFE** for key-presence validation because a missing key anywhere except the last position lets `jq -e` exit 0.

**Evidence (live tests):**

```
$ echo '{"removed":[],"renamed":[],"structural":[],"row_count_delta":0}' \
    | jq -e '.added, .removed, .renamed, .structural, .row_count_delta'
null
[]
[]
[]
0
exit: 0    ← BUG: 'added' is missing but exit 0 because last value (0) is truthy
```

```
$ echo '{"added":[],"removed":[],"renamed":[],"structural":[]}' \
    | jq -e '.added, .removed, .renamed, .structural, .row_count_delta'
[]
[]
[]
[]
null
exit: 1    ← OK: missing key happens to be last
```

**Canonical idiom — `has()` chained with `and` returns a single boolean, so `-e` can't be fooled by ordering:**

```
$ echo '{"removed":[],"renamed":[],"structural":[],"row_count_delta":0}' \
    | jq -e 'has("added") and has("removed") and has("renamed") \
              and has("structural") and has("row_count_delta")'
false
exit: 1    ← correctly fails
```

**Parse-failure path:** Invalid JSON makes `jq` exit 5 (program / parse error), distinct from exit 1 (key missing) — so the self-test can disambiguate.

**Citations:**
- `jq(1)` `--exit-status` / `-e` (line 261 of `man -P cat jq`): *"Sets the exit status of jq to 0 if the last output value was neither false nor null, 1 if the last output value was either false or null, or 4 if no valid result was ever produced."* — the wording `last output value` is what makes the comma-list approach unsafe.
- `jq(1)` Comma `,` operator (around line 540): *"If two filters are separated by a comma, then the same input will be fed into both and the two filters' output value streams will be concatenated in order."*
- `jq(1)` `has(key)` builtin (line 996): *"The builtin function has returns whether the input object has the given key, or the input array has an element at the given index."*

**Recommended idiom — single jq invocation that validates parse + all keys at once:**

```bash
# In tests/lint/catalog-diff.sh self-test:
output="$(./scripts/catalog-diff.sh --ref-diff v1.4.0 v1.4.1 --format=json)"

# Single jq call: validates the input is JSON AND all 5 keys are present.
# - Parse failure → jq exits 5
# - Any missing key → has() returns false → jq -e exits 1
# - All present → jq -e exits 0
if ! printf '%s\n' "$output" | jq -e '
    has("added")
    and has("removed")
    and has("renamed")
    and has("structural")
    and has("row_count_delta")
' >/dev/null 2>&1; then
    printf '✗ --format=json output missing required keys or not parseable\n' >&2
    printf '  output: %s\n' "$output" >&2
    exit 1
fi
```

**Why one call, not two:** `jq` parses the input as part of evaluating the filter. A single invocation validates parseability AND key presence simultaneously. A two-call pattern (`jq '.'` to parse, then `jq -e 'has(...)'` to validate keys) is strictly redundant — there's no input mutation between the calls and `jq` already does both checks.

### 4. ShellCheck rules for `scripts/catalog-diff.sh`

**Claim verified:** A minimal example using the patterns the script will need (`set -euo pipefail`, validate ref via `rev-parse`, read file via `git show "${ref}:${path}"`, count rows via `awk`) is **completely ShellCheck-clean** with no warnings at default severity, no warnings at `--severity=style`, and **no SC2016 suppressions needed**. The v1.4.1 hook scripts' `# shellcheck disable=SC2016` comments are specific to printf statements containing markdown code-fence backticks; `catalog-diff.sh` has no such pattern.

**Evidence — minimal example (matches the example in the question prompt):**

```bash

path="$2"

```


**Evidence — representative example (validate_ref + read_catalog_version + count_rows + awk + JSON emission):** ShellCheck triggered only `SC2034 (warning)` for two unused local variables in my prototype — a true-positive that the actual Dev will resolve by USING the variables (the prototype was incomplete).

**No SC2016 needed** because `catalog-diff.sh`'s printf statements emit JSON / text output (no literal markdown backticks). For comparison, the v1.4.1 hook scripts trigger SC2016 because they embed markdown code-fences:

```bash
# In session-end-handoff.sh line 102-103:
```

**The catalog-diff.sh script will only need SC suppressions if/when:**
- It emits markdown CHANGELOG output containing literal backticks → MIGHT need SC2016 in `--format=changelog` mode IF using printf with backtick-containing format strings. (Easy to dodge: use heredocs or `printf '%s'` with a variable holding the formatted text.)
- It uses `cat <<'EOF'` heredocs containing literal `$` (single-quoted heredoc disables expansion, but ShellCheck may flag) — SC2016 again.

**Recommended ShellCheck posture:**
1. Default severity (no `--severity=style`) is sufficient — matches the existing `.github/workflows/ci.yml` invocation (`shellcheck "$s"` with no severity flag).
2. Add suppressions ONLY at the line they're needed, with a comment explaining why (mirrors the v1.4.1 hooks pattern).
3. Run `shellcheck scripts/catalog-diff.sh` BEFORE committing; verify exit 0.
4. The CI lint job (lines 23-33 of `.github/workflows/ci.yml`) already runs ShellCheck on `scripts/*.sh` — `catalog-diff.sh` will be auto-discovered by the existing `for s in scripts/*.sh` glob; no CI changes required for this script.

### 5. CI parity for `--ci-check`

**Claim verified:** The current `catalog-version-sync` job (`.github/workflows/ci.yml` lines 79-99) uses `git diff --name-only origin/main...HEAD` (triple-dot — merge-base relative) to detect catalog and version changes. The Phase 1 Dev MUST replicate this **exact triple-dot syntax** in `--ci-check` mode, **NOT** use `git show` to read content, because `git show` fails with exit 128 when a path doesn't exist at the merge-base side (e.g. a PR that ADDS `.catalog-version` for the first time).

**Verbatim YAML body (lines 79-99 of `.github/workflows/ci.yml`):**

```yaml
catalog-version-sync:
  name: Catalog version bump check
  runs-on: ubuntu-latest
  if: github.event_name == 'pull_request'
  steps:
    - uses: actions/checkout@v4
      with:
        fetch-depth: 0

    - name: Check .catalog-version bumps when catalog changes
      run: |
        CATALOG_CHANGED=$(git diff --name-only origin/main...HEAD -- 'power-engineer/references/catalog/**/*.md' | wc -l | tr -d ' ')
        VERSION_CHANGED=$(git diff --name-only origin/main...HEAD -- 'power-engineer/.catalog-version' | wc -l | tr -d ' ')

        if [ "$CATALOG_CHANGED" -gt 0 ] && [ "$VERSION_CHANGED" -eq 0 ]; then
          echo "✗ Catalog files changed but .catalog-version was not bumped."
          echo "Files changed:"
          git diff --name-only origin/main...HEAD -- 'power-engineer/references/catalog/**/*.md'
          exit 1
        fi
        echo "✓ catalog-version sync OK"
```

**Logic decomposed:**

- **Trigger:** `pull_request` events only (`if: github.event_name == 'pull_request'`).
- **Range:** `origin/main...HEAD` — triple-dot syntax, which is `merge-base(origin/main, HEAD)..HEAD`. This compares from the merge-base (where the PR branched) to the current PR head, ignoring main-side changes that happened after the branch.
- **Detection mechanism:** `git diff --name-only` returns a count of changed files (additions, deletions, modifications all count uniformly). `wc -l | tr -d ' '` produces a clean integer.
- **Glob pattern (catalog files):** `'power-engineer/references/catalog/**/*.md'` (note the single quotes — pattern is interpreted by git, not the shell).
- **Glob pattern (version file):** `'power-engineer/.catalog-version'` (literal path).
- **Fail condition:** `CATALOG_CHANGED > 0 AND VERSION_CHANGED == 0`. Exit 1 with stderr listing the changed catalog files.
- **Pass conditions:** Either no catalog changes, OR catalog changes accompanied by a `.catalog-version` change (additions count).

**Edge cases the YAML handles:**

| Case | Behavior |
|------|----------|
| `.catalog-version` ADDED in PR (didn't exist on origin/main) | `git diff --name-only` reports the path → `VERSION_CHANGED=1` → passes |
| `.catalog-version` DELETED in PR | Same — `VERSION_CHANGED=1` → passes (though deletion is rare) |
| Catalog directory has subdirs (e.g. `cluster-a/`, `cluster-b/`) | `**/*.md` glob matches recursively |
| `INDEX.md` changes (no skill rows) | Counts as a catalog change — flags VERSION bump required (intentional: `INDEX.md` is auto-synced by `update-skill-count.sh` so this rarely fires alone) |
| Triple-dot vs double-dot if branch is force-pushed off main | `origin/main...HEAD` (triple-dot) computes merge-base dynamically — robust to force-pushes / rebases |
| Empty diff (no changes at all) | Both `CATALOG_CHANGED=0` and `VERSION_CHANGED=0` → passes (no fail condition triggered) |

**Recommended bash port (replicate this VERBATIM for `--ci-check` mode):**

```bash
ci_check() {
    # Replicates the catalog-version-sync logic from .github/workflows/ci.yml
    # lines 79-99. Exits 1 if catalog files changed without a .catalog-version
    # bump; exits 0 otherwise.
    local base_ref="${1:-origin/main}"

    local catalog_changed version_changed
    catalog_changed="$(git diff --name-only "${base_ref}...HEAD" -- \
        'power-engineer/references/catalog/**/*.md' | wc -l | tr -d ' ')"
    version_changed="$(git diff --name-only "${base_ref}...HEAD" -- \
        'power-engineer/.catalog-version' | wc -l | tr -d ' ')"

    if [[ "$catalog_changed" -gt 0 ]] && [[ "$version_changed" -eq 0 ]]; then
        printf '✗ Catalog files changed but .catalog-version was not bumped.\n' >&2
        printf 'Files changed:\n' >&2
        git diff --name-only "${base_ref}...HEAD" -- \
            'power-engineer/references/catalog/**/*.md' >&2
        return 1
    fi
    printf '✓ catalog-version sync OK\n'
    return 0
}
```

**Parity verification protocol (orchestrator must perform):**

1. Run the ORIGINAL YAML logic locally on a known-good PR commit (e.g. `git checkout v1.4.0`, then run the embedded shell verbatim against `origin/main...HEAD`).
2. Run `./scripts/catalog-diff.sh --ci-check` from the same checkout.
3. Compare exit codes AND stderr output line-by-line — they should be identical (modulo any line-buffering differences).
4. Construct a synthetic test case where catalog changes WITHOUT a version bump, verify both fail with exit 1 and the same error message.
5. Construct a synthetic test case where the PR ADDS `.catalog-version` (i.e. simulates the file being newly introduced) — verify both pass.

## Concrete recommendations for the Phase 1 Dev

These are the load-bearing instructions the `scripts/catalog-diff.sh` Dev must follow:

1. **Use this exact ref-validation idiom** in `validate_ref()`:
   ```bash
   git rev-parse --verify --quiet --end-of-options "${ref}^{commit}" >/dev/null
   ```
   The `--end-of-options` flag is mandatory for flag-injection defense; `--quiet` suppresses git's verbose stderr so the script can emit user-friendly errors.

2. **Use this exact jq idiom** in `tests/lint/catalog-diff.sh` `--format=json` self-test:
   ```bash
   jq -e 'has("added") and has("removed") and has("renamed") and has("structural") and has("row_count_delta")'
   ```
   **DO NOT use the comma-list pattern** (`jq -e '.added, .removed, ...'`) — it's silently buggy because `jq -e` only inspects the LAST value.

3. **For `--ci-check` mode, replicate the YAML's `git diff --name-only origin/main...HEAD` approach** — DO NOT use `git show` to read content from the merge-base side. The script must accept an optional base-ref argument (default `origin/main`) so it works in CI and local dev.

4. **No SC2016 suppression needed** unless the script emits printf format strings containing literal markdown backticks. The default ShellCheck posture (no severity flag) is sufficient — matches existing `.github/workflows/ci.yml` lint convention.

5. **Reuse `update-skill-count.sh`'s awk pattern** for catalog row counting (lines 9-22 of that file). It already handles both `| Skill |` and `| Suite |` headers, separator rows, and short-row filtering. Pipe `git show "${ref}:power-engineer/references/catalog/INDEX.md"` (and other catalog files) into the same awk program.

6. **Quote `git show` arguments correctly:** Always `git show "${ref}:${path}"` as a single double-quoted composite — never split. Suppress stderr (`2>/dev/null`) when probing optional content; do NOT suppress when validating required content (let the fatal error propagate).

7. **`.catalog-version` is unbumped in v1.4.2 scope** (per plan line 32) — the `--ci-check` self-test scenarios should include both "no catalog changes / no version bump → pass" AND "catalog changes / version unbumped → fail" cases.

## Flagged risks or surprises

### R1. The plan's Question 3 wording understates the jq danger

The plan text at line 112 reads:
> "Should the script invoke `jq` once or twice?"

This frames the question as a stylistic choice. **It's not** — the comma-list-with-`jq -e` pattern (which a Dev might naturally choose to avoid two calls) is silently buggy. The correct framing for the Phase 1 executor prompt is: *"Use a single `jq` call with `has(...) and has(...)` chain — never use comma-list with `-e`."* Suggest the orchestrator add this clarification when dispatching Task 1.4.

### R2. The plan's Question 5 Phase 1 Dev needs the EXACT YAML verbatim

The plan text at line 116 says "report its current logic verbatim" but doesn't surface the **edge cases** the YAML implicitly handles (file additions, file deletions, force-pushed branches, empty diffs). The Dev needs all of those documented for `--ci-check` to be at parity. This research doc fills that gap; Phase 1 dispatch should reference Section 5 above.

### R3. Performance consideration not in the plan

`git rev-parse --verify --quiet --end-of-options "${ref}^{commit}"` is ~4.6ms per call; `git show` is similar. The script will make ~10 subprocess calls per `--ref-diff` invocation (validate 2 refs × 1 call, read .catalog-version × 2, read CATALOG.md / INDEX.md × N catalog files × 2 sides). On the order of ~0.5–1.0 second total — acceptable for a CI step but slow for interactive use. **No optimization needed** for v1.4.2 scope, but flag this if future work adds an interactive `--watch` mode.

### R4. The plan correctly notes `jq` is a system dependency

Plan line 9 states: *"`jq` is invoked by `catalog-diff.sh --format=json` self-test (already a system dependency on macOS / Linux CI runners; `jq --version` is the install gate)."* — confirmed: macOS includes `jq 1.7.1-apple` by default at `/usr/bin/jq`, ubuntu-latest runners include `jq` in the base image. **No CI install step needed.** However, the self-test should `command -v jq >/dev/null 2>&1 || { echo "jq required for self-test" >&2; exit 2; }` as a defensive guard — a one-line addition.
