# v1.3.0 Behavioral Validation

**Context:** v1.3.0's plan explicitly deferred behavioral CI ("Out of Scope" →
"Claude-invoked behavioral CI: Deferred indefinitely — cost + model variance
don't justify inclusion in baseline PR CI"). Structural tests — `scanner-rules.sh`
running YAML rules against fixtures mechanically — were wired into CI. This doc
records the ONE-SHOT manual behavioral verification done at release time:
does Claude (not a bash runner) applying `scanner.md` + `detection-rules.yaml`
against each fixture produce the same detections as `scanner-rules.sh`?

## Method

`claude --bare -p` invocation per fixture, with:

- `--bare` — disables CLAUDE.md auto-discovery, auto-memory, hooks, plugin sync
- `--add-dir <modules>` — makes scanner.md + detection-rules.yaml readable
- `--disallowedTools 'Read(**/expected.md)'` — blocks reading the answer key
- `--model claude-sonnet-4-6` — pinned for reproducibility
- `ANTHROPIC_API_KEY` from environment (required by --bare)

Prompt (identical across all 5 fixtures):

> Read scanner.md and detection-rules.yaml from the added directory. Those are
> the scanner module and its detection rules. Apply those rules to the current
> working directory (the project being scanned). List EVERY detection in this
> exact format, one per line, nothing else — no preamble, no explanations, no
> headings:
>
> language: <name>
> framework: <name>
> sdk: <name>
> cloud_db: <name>
> <flag>: true
>
> Valid detection kinds: language, framework, sdk, cloud_db, monorepo,
> has_docker, has_terraform, has_ci_cd, has_tailwind. Emit nothing for kinds
> that don't apply. Do not invent detections. Do not guess.

Output is diffed against each fixture's `expected.md` after stripping
the markdown prefix from the `- **DETECT**: <line>` format.

|---|---|---|---|
| blank-project | (none) | (none) | MATCH |

**Score: 5/5 MATCH.**

## Limitations

- Manual one-shot, not wired into CI per plan's explicit deferral
- Model variance: another run on a different day / another Sonnet version may differ
- Training-data contamination: model knows what Next.js / Supabase / FastAPI look like generally — can't fully isolate scanner-module behavior from general knowledge
- Cost: ~$0.05–0.15 per 5-fixture run on Sonnet 4.6 (as of 2026-04-16)
- `--bare` mode requires `ANTHROPIC_API_KEY` env var (disables OAuth/keychain)

## Reproducing

See `scripts/run-behavioral-fixtures.sh`.

## Relationship to other tests

- **Structural** (`scanner-rules.sh`): verifies YAML rules mechanically against
  fixtures. Runs in CI every PR. Proves the rules are correct.
- **Behavioral** (this doc + the script): verifies Claude applying the module
  produces the same output as the mechanical runner. Manual-only.
- **End-to-end** (not done): a user running `/power-engineer` in a fresh
  Claude Code session against a real project. Not automated; won't be.
